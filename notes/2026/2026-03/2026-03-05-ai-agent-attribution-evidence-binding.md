# AI Agent 归因与证据绑定面试笔记

## 埋点数据归因是什么

埋点数据归因，本质是把用户行为日志和最终业务结果做贡献关联。

例如一个用户先看广告，再进官网，再看价格页，最后下单。归因要回答的是：这次转化到底主要算哪个触点的功劳，以及各触点分别贡献多少。

## AI Agent 架构设计

核心原则是：让模型做解释和编排，不做计算和拍结论。

- 数据层：负责埋点采集、数仓分层、完整性和一致性校验。
- 归因计算层：用版本化的 SQL 或 Python 脚本产出结构化结果。
- Agent 编排层：定时触发任务，按白名单调用内部工具和模型。
- 校验与人工闭环层：做规则校验、交叉校验和低置信度人工审批。
- 报告层：模板化生成报告，所有结论都必须绑定证据。

## 证据绑定怎么做

证据绑定的目标是：每一条结论都必须能追到具体数据、具体计算和具体版本。

先由归因脚本生成结构化 `evidence object`，例如：

```json
{
  "evidence_id": "ev_20260305_001",
  "metric": "signup_rate",
  "value": 0.127,
  "time_window": "2026-03-04",
  "sample_size": 182340,
  "table": "dws_signup_daily",
  "query_hash": "ab12cd34",
  "script_version": "attr_v3.2.1",
  "data_snapshot_id": "snap_20260305_09"
}
```

然后让模型只能引用 `evidence_id` 生成结构化 `claim`，而不是自由发挥：

```json
{
  "claim": "3 月 4 日注册转化率较前一日下降 12.7%",
  "evidence_ids": ["ev_20260305_001", "ev_20260305_002"],
  "confidence": 0.96
}
```

发布报告前要做强校验：

- 每条结论必须带 `evidence_id`。
- 结论中的数字必须和证据值一致。
- 同比、环比、占比、排名等计算由脚本完成，不交给模型心算。
- 保存 `run_id`、模型版本、prompt hash、脚本版本、数据快照 ID 和审批记录，保证审计与回放。

## 证据绑定和 RAG 引用的区别

- `RAG 引用`回答的是模型参考了哪些材料，重点是检索来源。
- `证据绑定`回答的是这条结论由哪份数据、哪次计算、哪个版本严格推出，重点是结论血缘。
- `RAG` 更适合制度文档、FAQ、产品说明这类非结构化知识。
- `证据绑定` 更适合报表、归因、风控、财务分析这类结构化场景。
- 在高准确场景里，RAG 可以用来检索规则文档和指标口径，但业务结论必须绑定到结构化证据对象。

## LangGraph 是什么

`LangGraph` 可以理解为一个面向生产的 Agent 编排框架。它把复杂任务抽象成共享状态、执行节点和路由边，让 Agent 不是靠一条长 prompt 硬跑，而是按图执行。

- `State`：保存消息、工具结果、中间结论和任务进度。
- `Node`：一个节点代表一步操作，比如调用模型、检索、执行工具或做校验。
- `Edge`：定义节点之间的跳转关系，可以是固定跳转，也可以是条件路由。
- `Runtime`：负责执行图、保存状态、恢复任务、流式输出和人工中断。

面试里可以把它概括成：`LangGraph 本质上是一个状态机加图运行时，LLM 只是图里的一个节点。`

## LangGraph 和 LangChain 的区别

- `LangChain` 偏高层，主要解决模型接入、工具封装、检索集成和快速搭应用。
- `LangGraph` 偏底层，主要解决状态管理、节点路由、持久化、中断恢复和多智能体编排。
- 简单场景优先用 `LangChain` 提速，复杂生产场景更适合直接用 `LangGraph` 做底层控制。
- 两者不是对立关系，很多 `LangChain` 的 Agent 能力本身就构建在 `LangGraph` 之上。

一句话总结：`LangChain` 偏易用，`LangGraph` 偏可控。

## LangGraph 和 MetaGPT 的区别

- `LangGraph` 是通用的 Agent 编排引擎，关注流程如何执行得可控、可恢复、可审计。
- `MetaGPT` 是更强调角色分工和 SOP 的多智能体协作框架，关注多个角色如何像团队一样协作。
- `LangGraph` 更适合生产级工作流、审批流和强治理场景。
- `MetaGPT` 更适合多角色协作、研发流程模拟和任务拆解型场景。

如果是上线系统，常见做法是借鉴 `MetaGPT` 的角色设计，但底层仍用 `LangGraph` 做运行时编排。

## LangGraph 的底层原理和流式响应

`LangGraph` 的底层本质是状态机加图执行引擎。每个节点读取当前 `state`，输出状态增量，再由边或条件路由决定下一步。运行时会在节点边界做持久化和检查点，所以任务可以中断、恢复、重放和人工介入。

流式响应不只是模型逐字输出，而是整个工作流的事件流：

- `messages`：流模型 token，适合前端逐字展示回答。
- `updates`：流节点级状态更新，适合展示当前执行到哪一步。
- `values`：流完整状态，适合调试或强观测场景。
- `custom`：流业务进度，例如检索进度、SQL 执行完成、归因分析完成。

工程上一般会把这些事件封装成统一结构，通过 `SSE` 或 `WebSocket` 推给前端，让用户看到的不只是“模型在打字”，而是整个 Agent 在实时执行。

## 多轮上下文怎么管理

多轮上下文一般不直接把全部历史消息塞进 prompt，而是做分层管理。

- 短期上下文：保留最近 `N` 轮原始消息，保证当前对话连贯。
- 任务状态：把用户目标、约束、已完成步骤和中间结论放进结构化 `state`。
- 历史摘要：对旧对话做压缩，只保留高价值信息，比如决策、事实和未解决问题。
- 长期记忆：只存稳定、可复用的信息，例如用户偏好、业务规则和历史结论。
- 检索式拼装：每次调用模型前，再动态组装系统提示词、最近消息、任务状态和检索出的长期记忆。

在 `LangGraph` 里，通常会把 `messages`、`summary`、`facts`、`task_state`、`artifacts` 分开管理。这样可以同时控制 token 成本、保持语义连续，并降低无关历史对当前推理的干扰。

## Agent 汇总结论不准怎么办

这个问题一般不能只靠调 prompt 解决，通常要从输入、汇总和校验三层一起处理。

- 先约束输入：不要把大段原始上下文直接喂给汇总节点，而是先整理成结构化中间结果，例如 `finding`、`evidence_id`、`confidence`、`source`。
- 先算后说：数值、排序、对比、归因结果由脚本或工具先算好，汇总 Agent 只负责组织语言，不负责自己推数字。
- 增加 `verifier` 或 `reviewer` 节点：检查结论是否被证据支持、数字是否一致、是否遗漏冲突信息。
- 做多路交叉验证：可以让两个 summarizer 独立汇总，再让 reviewer 比对，不一致就回退到重检索或重跑工具。
- 允许输出不确定性：当证据不足时，不强行给确定结论，可以输出待确认项或建议人工复核。
- 设计失败回退：一旦校验失败，就降级为证据列表、初步判断和待确认问题，而不是直接生成最终答案。

面试里可以概括为：`结构化输入 + 先算后说 + verifier 校验 + 多路交叉 + 失败回退`。

## Human-in-the-loop 怎么形成闭环

`Human-in-the-loop` 不只是中间插一个人工审批节点，而是要形成“触发、介入、记录、回写、优化”的完整闭环。

- 先定义触发条件：例如低置信度、多 Agent 结论冲突、高风险任务、关键数据缺失、超出规则边界。
- 给人工明确操作位：允许人工通过、驳回、修正结论、补充约束，或者指定更可信的数据源和工具。
- 保留审计链：保存 `run_id`、原始输入、中间状态、模型输出、人工修改内容和最终结果。
- 把反馈写回系统：沉淀到规则库、提示词模板、路由策略和评测集，避免同类错误反复出现。
- 定期优化阈值和策略：观察人工介入率、驳回率、重跑率和最终正确率，持续调整系统。

面试里可以概括为：`触发介入 + 审计留痕 + 反馈回写 + 持续优化`。

## 算法题

无序数组中找不重复的最小值，可以先统计每个元素出现次数，再找出现次数为 `1` 的最小值。

```python
from collections import Counter

def min_unique(nums):
    counts = Counter(nums)
    answer = None
    for value, count in counts.items():
        if count == 1 and (answer is None or value < answer):
            answer = value
    return answer
```

时间复杂度是 `O(n)`，空间复杂度是 `O(n)`。
