    模拟面试记录：字节跳动 AI代码分析安全研发实习生¶

    日期：2026-03-10 目标岗位：AI代码分析安全研发实习生 - 安全与风控（ByteIntern） 面试形式：技术+行为混合 拷打强度：5/5 面试官设定：字节安全与风控部门 Tech Lead

    Q1: HarnessAgent 项目¶

    面试官：你简历上写了 HarnessAgent——基于 LangGraph 构建 AI 代码漏洞分析与 harness 生成智能体。讲一下核心思路：它要解决什么问题，怎么设计的，你具体做了什么。

    我的回答：这个 harness agent 是一个自动的针对开源软件项目自动生成可编译可运行、能有效提高测试覆盖率的 fuzz driver。使用了基于 LangGraph 框架的自动修复 agent，包含若干代码分析工具，对 11 个软件项目每个选择 300 个函数进行生成，最终发现了 15 个真实 bug。

    追问 1：自动修复 loop 怎么设计的？¶

    我的回答：给 agent 提供查看函数定义、查找 symbol 的工具，基于 clangd LSP，也用了 tree-sitter 补充。agent 拿到编译报错信息，修复成功就退出，最多尝试 6 轮。编译错误做了分类，简单的 rule-based 改，复杂的丢回 agent。

    追问 2：错误分类的边界？¶

    我的回答（自我纠正）：不对，其实分成了链接 error、include error、其他语法错误、docker 错误。链接 error 需要修改编译脚本，include 可以基于规则解析路径改编译参数，语法错误 agent 自己改。

    面试官点评：前后说法差距大，面试官能感觉到对项目记忆不牢。

    追问 3：链接错误怎么处理？¶

    我的回答：我们没有解决得很好，采用了偷懒的方法——每个项目已有几个写好的编译脚本，就换着试。

    面试官点评：坦诚，比硬编答案强。

    Q2: 开源软件漏洞的自动化跨版本迁移¶

    面试官：这个项目的动机是什么？为什么需要跨版本漏洞迁移？实际应用场景是什么？

    我的第一次回答（答错题）：如果现在让我来做，我会让 agent 自己来改并修复，因为 model 越来越强了。

    面试官：我问的不是 HarnessAgent 的链接错误。我问的是跨版本迁移的动机。

    我的回答：我们想做的是一个版本的软件同时具备尽可能多的历史 bug，作为 fuzzing 的 benchmark。真实的 bug，真实的 PoC。

    追问：最大技术难点？patch revert 跨版本不适用怎么办？¶

    我的回答：拿到 bug 版本的函数执行 trace，把这些函数放在无 bug 版本中，进行 build error 修复。这个过程很困难——不同版本间 API、常量、结构体定义变化很大。最初用 rule-based 做效果不好，最近两个月改成 agent 做，好一些但还没完全解决。

    追问：agent 怎么知道旧 API 在新版本对应什么？¶

    我的回答：提供给 agent 两个版本的源码和一个代码数据库，用 libclang 分析两个版本的代码，得到所有 symbol 的准确行号。LLM 有了行号就可以去查源码得到定义。

    追问：为什么 HarnessAgent 用 clangd，这个项目用 libclang？¶

    我的回答：我自己做的时候先试了 tree-sitter 对 C 没法用，选了 libclang。同学选了 clangd。clangd 好处是可以增量查询每次耗时小，我的 libclang 是一次性全量解析，第一次运行会很久。

    Q3: 静态分析基础 + 系统设计¶

    面试官：AST 层面分析和 IR 层面分析各自能做什么、不能做什么？做代码安全检查怎么选？

    我的回答：AST 保留源码信息更多，可以获得行号，适合给 LLM 看（训练了更多源码而非 IR）。IR 经过优化，获得源码信息需要 debug info 有损失，但 IR 是 SSA 范式适合做各种静态分析和优化。

    面试官纠正：说"IR 多少进行过优化"不够准确，-O0 和 -O2 的 IR 差别很大。

    追问：给字节大型 C++ 仓库做安全检查工具（UAF/double free/buffer overflow），怎么设计？¶

    我的回答：可以用 AI 辅助 fuzzing。我最近用 Claude Code 为已经被 fuzz 很久的开源软件生成 fuzz driver，发现了几个 bug 拿到一个 CVE。如果用静态方法的话肯定是在 IR 层面，会调研市面上最好的静态分析工具试试。

    面试官点评：fuzzing 方案有说服力，但静态分析"调研一下试试"不是答案。面试官听得出来。

    Q4: 行为题——推翻技术方案¶

    面试官：有没有碰到技术方案走不通需要推翻重来的情况？

    我的回答：跨版本迁移项目，最开始用 rule-based 修复编译 error，但 C 语言报错很复杂永远做不完，就把代码删了和导师说想用 agent 试试，现在看来有希望解决。

    追问：做了多久意识到走不通？¶

    我的回答：大概半年。第一个开源项目迁移成功，但第二个完全走不通，就知道坏了。

    面试官点评：半年不短，第一个项目上过拟合了。

    追问：如果再来一次怎么更早发现？¶

    我的回答：项目初期就要和更多人讨论，同时敏锐观察市场上的新技术比如 agent，当时反应迟钝了。

    自评环节¶

    • 我的自评：Hire
    • 最好的回答：跨版本迁移项目讲得比较清楚
    • 最差的回答：静态分析知识准备不充分
    • 面试官评价：Mixed 偏 Hire

    面试官内心独白¶

    简历很亮——USENIX Security、编译器安全、在读博，背景比大多数实习生强。但面试一开始讲 HarnessAgent，一分钟了我还不知道项目怎么工作的。

    错误分类前后矛盾让我怀疑他对项目的掌握程度。跨版本迁移答错题说明注意力不够集中或太紧张。libclang 那段是全场最技术的回答。

    静态分析是分水岭。fuzzing 有说服力有 CVE，但静态方案说"调研试试"——岗位叫 AI 代码分析，如果 CodeQL、Infer、Coverity 一个都讲不出来，会担心知识面。

    总体：技术底子有，项目真实，但表达是硬伤。只看简历 Strong Hire，面试完 Mixed 偏 Hire。实习生可以过，但得补静态分析。

    Top 3 改进建议¶

    1. 练习结构化表达（最重要）¶

    每个项目准备 2 分钟版本，结构固定：问题 → 方案 → 你的角色 → 关键技术决策 → 结果数字。先说框架再填细节。

    2. 补静态分析知识¶

    至少了解： - CodeQL（GitHub 在用） - Semgrep（规则型） - Infer（Facebook，过程间分析） - Coverity

    能说出原理差异和适用场景。

    3. 行为题给具体方法论，不要给感悟¶

    • ❌ "多和人讨论"
    • ✅ "现在每两周和导师做一次 milestone check，如果连续两周没有在新数据上验证，就强制停下来评估方案可行性。"

    第二轮模拟面试记录¶

    时间：同日，第一轮结束后立即重来 改进目标：结构化表达、静态分析知识、数字准确性

    Q1: 跨版本漏洞迁移（自选项目）¶

    面试官：挑一个最能代表你技术能力的项目，两到三分钟讲清楚：解决什么问题、怎么做的、你负责什么、结果怎样。

    我的回答：自动化漏洞迁移项目是将一个 C 项目的历史 bug 尽可能多的集中到同一个版本上，构造 bug 数据集作为 fuzzer 的 benchmark。做法是收集 bug 版本触发 bug 时走过的函数路径，找到在新版本里被修改过的函数——一个函数在 bug 触发时执行过且被修改了，结果 bug 在新版本不触发，我就认为它可能和 bug 触发有关系。所以选择把 bug 版本的函数直接复制到新版本里。这样的 patch 肯定编译不通过（API、结构体定义、常量都会变），用 LLM agent 做编译错误修复。修复后用原 PoC 测试复现，不能复现就用 target fuzz 以 bug 触发时的 stack 为目标 fuzz。随后做 patch 最小化，有利于多 bug 迁移到同一版本不冲突。项目都是我自己做的。目前在 opensc 中完成端到端验证，生成了一个最多 19/26 个 bug 的版本。

    面试官评价：比第一轮好非常多。问题、方法、结果都一次性讲清了。

    追问 1：怎么判断函数被修改过？¶

    我的回答：就是 diff。没有区分函数具体修改了什么，因为最后会做最小化来去噪。

    面试官：先粗筛再最小化去噪，策略说得通。代价是编译修复工作量变大。最终最小化掉了多少比例？

    我的回答：没有仔细统计。在 opensc 中许多 bug 最后只改一个函数就能触发，最初可能有十几个函数可能相关。

    面试官建议：这个数据统计好了面试时直接说，很有说服力。

    追问 2：agent 修复成功率？有没有手动介入？¶

    我的回答：19 个成功迁移的 agent 都是成功的。开发过程中我希望 agent 都能修复，花了很多时间修改 prompt 和 tool。不会考虑手动介入。

    面试官：通过迭代 prompt 和工具提升能力而不是自己兜底——做工具不做人力，思路对。

    追问 3：26 个里没成功的 7 个卡在哪？¶

    我的回答（数字澄清）：26 个历史 bug 中 14 个是这个版本本来就能触发的。我尝试迁移了 12 个，其中 9 个成功触发，3 个无法触发。9 个之间存在冲突（依赖修改相同函数），合并时保留了 14+5=19 个。

    面试官点评：实际迁移贡献是 5 个，和"19/26"的第一印象差别很大。这个数字面试里要主动讲清楚。

    追问 4：3 个无法触发的原因？¶

    我的回答：编译通过了但 PoC 跑不出来。做了 target fuzz 依然没触发。可能是有些 bug 依赖的东西不在函数范围里（比如结构体），agent 修复时会改变代码导致不能触发。

    面试官点评：假设合理，但自己承认"没有去分析"。作为项目主导者，失败 case 的 root cause 分析应该做。

    Q2: HarnessAgent + SanCov 实现¶

    面试官：HarnessAgent 和跨版本迁移都用了 LLM agent 修编译错误，两个项目的 agent 设计有什么不同？

    我的回答：这个项目不是我主导的，我参与了讨论和一个代码覆盖率的实现。两个项目 agent 修复的复杂程度不一样，但结构基本相似——看编译错误报告、解析错误、agent 调用工具修复、再编译的循环。针对的问题不同，工具设计、错误解析、prompt 都不同。比如 bug 迁移需要同时提供两个版本的代码。

    追问：代码覆盖率实现具体是什么？¶

    我的回答：项目为给定函数生成 fuzz driver，driver 里不仅有目标函数还有结构体初始化等。我们需要只统计从目标函数进去走过的基本块数量。实现基于 LLVM 的 sanitizer coverage——编译器维护一个小的内存区域，每个基本块对应一个 byte，值表示走过次数。我们在执行目标函数前清零，之后读取这个区域，就实现了只统计目标函数覆盖率。

    面试官评价：SanCov inline 8-bit counters，入口清零出口读取——干净利落。说明对 LLVM sanitizer 底层机制有了解。全场最硬的技术回答。

    Q3: 系统设计——CI 自动安全扫描¶

    面试官：假设来字节实习，第一个任务是给几百万行 C/C++ 仓库加自动化安全检查，集成到 CI，怎么设计？重点考虑怎么让它实际用起来。

    我的回答：用 CodeQL 来做。CodeQL 可以查询 code pattern，适合大型项目。让 AI 生成 query，每次添加新代码执行 query 扫描。速度快，不会像每个函数都给 AI 那么慢，也能有效发现常见问题。

    追问 1：AI 生成的 query 怎么保证质量？误报怎么办？¶

    我的回答：可以用多个模型生成多次，互相打分选出最优的。生成的结果也需要 AI 先看，再传给人工。

    面试官挑战：AI 生成 query，AI 再审核——如果审核本身也不靠谱？除了 AI 查 AI，有没有更硬的验证手段？

    我的回答：用项目历史上已知的 bug 进行测试，能找到真实 bug 并且误报率低的留下。

    面试官评价：用已知 bug 做 ground truth 验证——本质上是测试集驱动的质量保障，比 AI 查 AI 靠谱得多。

    Q4: 动机题¶

    面试官：在犹他大学读博但投字节实习，怎么考虑的？

    我的回答：现在每个人都要实习，比在学校学到更多。长期准备毕业回国工作，也是为以后做准备。

    追问：能实习多久？怎么保证投入度？¶

    我的回答：可以线下实习 4 个月，和导师商量过。最近在给之前的事情收尾，暑假可以接着干。

    自评环节¶

    • 我的自评：Hire
    • 最好的回答：跨版本迁移项目讲得清楚
    • 最差的回答：项目具体效果数字没准备好，不知道失败原因
    • 面试官评价：Hire

    面试官内心独白¶

    开场讲跨版本迁移，两分钟讲完我就知道这个人准备过了——问题清楚、pipeline 完整、有结果数字。比大多数实习生强。

    19/26 那个数字追问才发现实际是 14+5。没有故意夸大，但没有主动讲清楚。

    SanCov 那段是加分项。inline 8-bit counters、清零读取——不只是调 API，是真的理解编译器插桩的底层机制。这个能力在安全研发岗很值钱。

    系统设计给了 CodeQL + AI 生成 query 的方案，方向对但展开不够。想听到更完整的故事：query 生成 → 增量扫描 → 结果过滤 → 开发者看到什么 → 误报怎么反馈改进 query。不过对实习生，能想到 CodeQL 并用已知 bug 验证质量已经不错了。

    3 个失败 case 没分析是减分的。如果能说"我分析了，2 个因为 X，1 个因为 Y，方法在 Z 条件下有局限"——Credibility 直接拉满。

    总体 Hire。项目真实、技术底子好、表达清晰。主要短板是边界 case 和失败场景缺乏主动分析习惯。

    Top 3 改进建议（第二轮）¶

    1. 数字讲准确、讲主动¶

    不要说"19/26"让面试官自己理解。直接说："14 个历史 bug 本来就能触发，我额外迁移成功了 5 个，总共 19 个。"

    2. 分析失败 case，带到面试里¶

    3 个无法触发的 bug——回去花一两小时分析 root cause。面试时主动讲："有 3 个没成功，原因是 X。方法在 Y 场景下有局限，未来可以通过 Z 改进。"

    3. 系统设计题先给框架再填细节¶

    被问到"怎么设计"时，先花 15 秒给全景："分三步——第一步 X，第二步 Y，第三步 Z。"然后再展开。面试官知道你有全局思考。
